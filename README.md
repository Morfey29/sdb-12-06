### Задание 2

Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц: 

- пользователи, 
- книги, 
- магазины (столбцы произвольно). 

Опишите принципы построения системы и их разграничение или разбивку между базами данных.

*Пришлите блоксхему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.* 

![alt text](https://github.com/Morfey29/sdb-12-06/blob/main/imgage/7.png)

Описание схемы:

Кластер Users: Хранит таблицу пользователи. Использует master-slave репликацию для распределения нагрузки чтения.
Кластер Shops: Хранит таблицу магазины. Также использует master-slave.
Кластер Books: Хранит таблицу книги.

Вертикальное разделение: Весь кластер отвечает только за книги.
Горизонтальное разделение: Внутри кластера данные разбиты на 3 шарда (по id книги). В каждом шарде своя master-slave репликация для надежности и производительности чтения.
Шардинг-прокси: Это ключевой элемент. Он принимает все запросы от приложения, знает правила шардинга (какой id книги на каком шарде лежит) и направляет запросы на нужные серверы. Для запросов, требующих данные с разных шардов (например, поиск по автору), прокси может выполнить распределенный запрос и собрать результаты.

Вертикальный шардинг (Vertical Sharding)

При вертикальном шардинге мы разделяем таблицы по разным серверам. Это логичное разбиение, когда таблицы слабо связаны друг с другом.

Принцип: Каждая таблица целиком живет на своем выделенном сервере баз данных.
Схема:

Сервер А (Users DB): Хранит таблицу пользователи.
Сервер Б (Books DB): Хранит таблицу книги.
Сервер В (Shops DB): Хранит таблицу магазины.
Режим работы: Все серверы работают в обычном режиме, принимая как запросы на чтение, так и на запись для своих таблиц.
Проблема: Если нам понадобится выполнить JOIN между таблицами магазины и книги (например, "найти все книги в магазине"), сделать это напрямую будет невозможно, так как данные на разных серверах. Придется делать запросы на уровне приложения.
Горизонтальный шардинг (Horizontal Sharding)

При горизонтальном шардинге мы разделяем данные внутри одной таблицы по разным серверам. Это полезно, когда одна таблица становится огромной. Выбирается ключ шардирования (shard key), по которому данные распределяются.

Возьмем для примера таблицу книги. Допустим, их миллионы.

Принцип: Книги разбиваются на части по какому-то правилу и хранятся на разных серверах.
Вариант схемы (по диапазону):

Сервер Г (Books Shard 1): Хранит книги с id от 1 до 1 000 000.
Сервер Д (Books Shard 2): Хранит книги с id от 1 000 001 до 2 000 000.
Сервер Е (Books Shard 3): Хранит остальные книги.
Режим работы: Каждый сервер работает в обычном режиме. Приложению нужен специальный механизм (proxy или логика в коде), чтобы понять, на какой сервер отправить запрос. Например, запрос книги с id = 500 пойдет на Сервер Г, а с id = 1 500 000 — на Сервер Д.
Проблема: Сложность реализации. Поиск книг по автору, если они разбросаны по всем шардам, потребует опроса всех серверов и агрегации результатов.
Комбинированный подход и блок-схема

На практике часто используют комбинацию подходов. Для нашей системы можно предложить следующее:

Вертикальное разделение: Разносим таблицы пользователи, книги и магазины на отдельные кластеры серверов, чтобы нагрузка от операций с разными сущностями не пересекалась.
Горизонтальное разделение: Самую большую и нагруженную таблицу, например книги, дополнительно шардируем горизонтально внутри ее собственного кластера.
