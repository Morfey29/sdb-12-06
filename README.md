### Задание 2

Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц: 

- пользователи, 
- книги, 
- магазины (столбцы произвольно). 

Опишите принципы построения системы и их разграничение или разбивку между базами данных.

*Пришлите блоксхему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.* 

![alt text](https://github.com/Morfey29/sdb-12-06/blob/main/image/7.png)

Описание схемы:

Кластер Users: Хранит таблицу пользователи. Использует master-slave репликацию для распределения нагрузки чтения.
Кластер Shops: Хранит таблицу магазины. Также использует master-slave.
Кластер Books: Хранит таблицу книги.

Вертикальное разделение: Весь кластер отвечает только за книги.
Горизонтальное разделение: Внутри кластера данные разбиты на 3 шарда (по id книги). В каждом шарде своя master-slave репликация для надежности и производительности чтения.
Шардинг-прокси: Это ключевой элемент. Он принимает все запросы от приложения, знает правила шардинга (какой id книги на каком шарде лежит) и направляет запросы на нужные серверы. Для запросов, требующих данные с разных шардов (например, поиск по автору), прокси может выполнить распределенный запрос и собрать результаты.

Вертикальный шардинг (Vertical Sharding)

При вертикальном шардинге мы разделяем таблицы по разным серверам. Это логичное разбиение, когда таблицы слабо связаны друг с другом.

Принцип: Каждая таблица целиком живет на своем выделенном сервере баз данных.
Схема:

Сервер А (Users DB): Хранит таблицу пользователи.
Сервер Б (Books DB): Хранит таблицу книги.
Сервер В (Shops DB): Хранит таблицу магазины.
Режим работы: Все серверы работают в обычном режиме, принимая как запросы на чтение, так и на запись для своих таблиц.
Проблема: Если нам понадобится выполнить JOIN между таблицами магазины и книги (например, "найти все книги в магазине"), сделать это напрямую будет невозможно, так как данные на разных серверах. Придется делать запросы на уровне приложения.
Горизонтальный шардинг (Horizontal Sharding)

При горизонтальном шардинге мы разделяем данные внутри одной таблицы по разным серверам. Это полезно, когда одна таблица становится огромной. Выбирается ключ шардирования (shard key), по которому данные распределяются.

Возьмем для примера таблицу книги. Допустим, их миллионы.

Принцип: Книги разбиваются на части по какому-то правилу и хранятся на разных серверах.
Вариант схемы (по диапазону):

Сервер Г (Books Shard 1): Хранит книги с id от 1 до 1 000 000.
Сервер Д (Books Shard 2): Хранит книги с id от 1 000 001 до 2 000 000.
Сервер Е (Books Shard 3): Хранит остальные книги.
Режим работы: Каждый сервер работает в обычном режиме. Приложению нужен специальный механизм (proxy или логика в коде), чтобы понять, на какой сервер отправить запрос. Например, запрос книги с id = 500 пойдет на Сервер Г, а с id = 1 500 000 — на Сервер Д.
Проблема: Сложность реализации. Поиск книг по автору, если они разбросаны по всем шардам, потребует опроса всех серверов и агрегации результатов.
Комбинированный подход и блок-схема

На практике часто используют комбинацию подходов. Для нашей системы можно предложить следующее:

Вертикальное разделение: Разносим таблицы пользователи, книги и магазины на отдельные кластеры серверов, чтобы нагрузка от операций с разными сущностями не пересекалась.
Горизонтальное разделение: Самую большую и нагруженную таблицу, например книги, дополнительно шардируем горизонтально внутри ее собственного кластера.

Преимущества различных методов масштабирования и обеспечения отказоустойчивости
1. Активный master-сервер и пассивный репликационный slave-сервер

Это классическая схема master-slave с одним ведомым. Основные преимущества
   - Отказоустойчивость: при сбое master-сервера можно быстро переключить нагрузку на slave (вручную или автоматически), минимизируя простой.

   - Балансировка нагрузки чтения: запросы на чтение (SELECT) можно направлять на slave, снижая нагрузку на master, который обрабатывает все записи.

   - Резервное копирование в реальном времени: slave всегда содержит актуальную копию данных, что упрощает создание резервных копий без остановки master.

   - Простота настройки и обслуживания: схема хорошо документирована и реализована в большинстве СУБД.

2. Master-сервер и несколько slave-серверов

Расширение предыдущего подхода с несколькими репликами даёт дополнительные выгоды:

   - Масштабирование чтения: чем больше slave, тем выше совокупная производительность системы на операциях чтения, что критично для проектов с высокой долей запросов на выборку.

   - Географическая распределённость: slave можно разместить в разных дата-центрах для приближения данных к пользователям и повышения доступности при региональных сбоях.

   - Специализация реплик: разные slave можно использовать для разных задач — например, один для аналитических отчётов, другой для резервного копирования, третий для тестирования новых запросов.

   - Устойчивость к отказам реплик: выход из строя одного slave не влияет на работу системы в целом, достаточно оставшихся.

3. Активный сервер со специальным механизмом репликации — DRBD (Distributed Replicated Block Device)

DRBD работает на уровне блочных устройств, реплицируя данные между серверами синхронно или асинхронно. Его преимущества:

   - Высокая надёжность данных: синхронный режим гарантирует, что данные записаны на оба узла до подтверждения операции, что исключает потери при сбое основного сервера.

   - Прозрачность для приложений: репликация выполняется на уровне диска, поэтому приложения могут использовать стандартные файловые системы, не требуя изменений.

   - Быстрое переключение при отказе: в паре с менеджером кластера (например, Pacemaker) DRBD обеспечивает автоматическое переключение на резервный узел за секунды.

   - Эффективность для баз данных: позволяет строить высокодоступные кластеры для любых СУБД, включая те, что не имеют встроенной репликации.

4. SAN-кластер (Storage Area Network)

SAN представляет собой выделенную высокоскоростную сеть хранения данных, к которой подключаются несколько серверов. Преимущества:

   - Централизованное управление хранилищем: все данные находятся на надёжных дисковых массивах, что упрощает администрирование, резервное копирование и расширение ёмкости.

   - Высокая производительность и отказоустойчивость: SAN обычно строятся с избыточными компонентами (контроллеры, каналы связи), обеспечивая бесперебойный доступ.

   - Гибкость масштабирования: можно добавлять новые серверы, использующие общее хранилище, без переноса данных; также легко наращивать ёмкость хранилища.

   - Поддержка продвинутых функций: снапшоты, удалённая репликация, дедупликация, шифрование на уровне массива.

   - Консистентность данных: все серверы видят одно и то же состояние данных, что важно для кластерных конфигураций (например, Oracle RAC).


Каждый из этих методов подходит для разных сценариев: master-slave — для распределения нагрузки чтения и простого резервирования, DRBD — для построения отказоустойчивых кластеров с минимальным временем восстановления, SAN — для централизованного высокопроизводительного хранения в крупных инфраструктурах.
